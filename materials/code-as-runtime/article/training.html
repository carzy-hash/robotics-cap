Code as Runtime · Runtime Trace RL · 03

在 Runtime 中 rollout。
沿完整 trace 做 RL。

Code as Runtime 的 RL 对象不是单步 action，也不是一段脱离执行的文本，而是主 Agent 与 runtime 共同产生的可执行轨迹：读取 observation、选择 decision、写代码、调用工具、修改 Action Queue、检查结果并在失败后恢复。它借鉴可执行 code-agent trace[3]；CaP-X 已进一步展示具身 coding agent 的多轮反馈与 verifiable-reward RL[8]。本文在此基础上讨论完整 runtime trace 的 outcome、progress、约束与成本信号。

01 · Optimization object

训练对象是一条可执行 trace，
不只是最后一段动作。

它更像可执行 Code Agent 的训练闭环：模型自回归写出下一个 runtime decision，环境执行代码、工具与动作，新的 observation 再回到上下文。这种 interleaved reasoning / acting 形式可追溯到 ReAct[2]；RL 优化的是这条跨越多轮的闭环策略，而不只是最终动作是否碰巧成功。

WHAT IS AUTOREGRESSIVE

整个主策略接口

每一轮输出都接在已有 trace 后面：观察请求、代码、工具结果、queue 决策、自检与恢复选择都成为下一 token 的条件。

WHAT IS NOT IMPLIED

不要求逐 token 控制电机

Action Chunk 仍可由 ACT 类策略[4]、π₀ 类 flow policy[5]、规划器或插值器生成；自回归对象是上层 runtime trace。

FAIR VLA COMPARISON

区别在主要监督出口

RT-2 等 VLA 也可以自回归地产生动作 token[13]。这里真正不同的是：Code as Runtime 把可执行策略过程本身作为主要生成与 RL 对象，而不是只把奖励落在动作序列上。

02 · Verifiers and credit

最终成功给方向，
过程奖励给信用分配。

单一 success reward 太稀疏，也容易让策略找到仿真捷径。更合适的做法是保留独立 outcome verifier，同时用可信度不同的检查器刻画进度、约束、调用和恢复质量。

R(trace)=outcome+Δ progress+endpoint checks−violations−runtime cost
信号监督什么优先证据主要风险
Outcome任务最终是否完成仿真状态 / 真机传感器 / 人工复核稀疏，可能掩盖错误过程
Progress距离子目标是否真实缩短potential difference、阶段条件、几何量被代理指标劫持
Endpoint目标、观察、约束、调用、状态、恢复规则、物理、校准后的视觉 verifier错误检查器污染信用分配
Safety碰撞、超力、越界和权限确定性规则与独立安全系统不应由模型自评覆盖
Cost无效观察、冗长代码、频繁改 queuetoken、延迟、调用和执行计数过强会压制必要恢复
Progress reward 最好描述状态变化，而不是奖励“说自己取得了进展”。

例如奖励夹爪到可达 pre-grasp 的势能差、物体进入目标区域的几何变化，或经过校准的接触状态变化；自然语言 self-check 只能作为待验证声明，不能自己给自己打分。

03 · Rollout and optimization loop

每轮 RL 都要经过执行，
而不是只批改生成文本。

一轮训练从可重放的 runtime state 开始，采样 Agent decision，执行产生真实 observation，再由独立 verifier 写回奖励与失败端点。SFT 和 privileged solver 只提供初始状态分布；主要数据随后应逐步转向 on-policy rollout。

00
INITIAL STATE

从可重放 runtime snapshot 开始

加载任务、因果 observation、Action Queue、active handlers 与最近事件；状态接口和部署时一致。

POLICY SAMPLE

自回归生成下一次 decision

采样 CONTINUE、analysis、代码、action call 或 handler update，而不是直接读取 oracle action。

让输出真正产生后果

运行代码、消费 Action Queue、触发 handler，并把 stdout、异常、传感器与进度写回 trace。

STEP

VERIFIER PASS

在结构化端点判分

独立检查 outcome、状态进展、安全约束、调用有效性、恢复质量与 runtime cost。

把奖励落到相关决策

结合 terminal return、progress difference 和端点标签，区分观察错误、代码错误与执行失败。

04 · Cold start: privileged projection

可以借用真值完成任务，
不能伪造真值的来源。

在策略尚不能稳定 rollout 时，仿真 GT 可以提供可行解与检查标签，作为 RL 之前的冷启动手段。但如果 trace 声称视觉系统推导出了它从未看到的精确量，就会制造不可部署的因果捷径。正确目标是 deployment-shaped initialization，而不是让 oracle 长期替代 on-policy 探索。

PRIVILEGED ORACLE

从 mesh GT 求解抓取

mesh_gt = sim.object_mesh(id)
pose_gt = sim.object_pose(id)
g_star = antipodal_grasp(mesh_gt, pose_gt)
execute(g_star)

它负责回答“存在什么可行解”，并提供碰撞、接触与最终成功标签。

→
DEPLOYMENT-SHAPED TRACE

从视觉表征产生候选

mesh_hat = reconstruct(multiview_rgbd)
g_vis = propose_grasp(mesh_hat)
residual = refine_grasp(obs, g_vis)
queue.push(grasp_chunk(g_vis + residual))

主 Agent 只调用部署时存在的 perception 与 refinement helper。

→
TEACHER RESIDUAL

Magic Offset 只做脚手架

delta_magic = g_star - g_vis
label(refine_grasp, delta_magic)
record(privileged=true,
       residual=delta_magic,
       projection_error=...)

offset 用来训练可部署的 residual helper，并在课程中逐步归零；它不能被冒充成视觉观测结果。

策略输入只包含 sim/real 交集中的 observation、状态和 API。
Teacher sidecar单独保存 GT、Magic Offset、oracle action、失败原因和标签置信度。
验收条件撤掉 sidecar 后重新 rollout；只有 observation-only 策略仍通过 verifier 才进入下一阶段。

同一个投影办法可以扩展到更多任务

PLACEMENT稳定放置

GT 支撑面与稳定姿态产生 oracle goal；视觉 scene reconstruction 产生候选，residual teacher 监督最终 pose refinement。

ARTICULATION门与抽屉

GT joint axis 和接触点产生可行轨迹；学生从视觉与力反馈估计关节状态，并学习何时更新 force handler。

OCCLUSION主动观察

GT visibility graph 选择最佳相机位；投影 trace 改写为根据不确定度调用换视角与多帧融合。

RECOVERY失败恢复

仿真直接给出 slip、collision 或 missed grasp 原因；学生只看到可观测症状，并学习重新观察、回撤或停止。

05 · Runtime action space

RL 不只探索“做什么”，
也探索 runtime 应该怎样写。

同一任务可以有多种正确 Action 输出和 handler 实现。rollout 可以采样候选代码、执行单元测试、注入异常，再由 verifier 比较成功率、恢复能力和运行成本，从而避免主 Agent 被固定在一种僵硬模板里。

ACTION BUILDER CANDIDATES

正常任务动作

# equivalent valid implementations
queue.push(interpolate(keyframes))
queue.push(plan_to_pose(goal))
queue.push(vla_primitive(goal, horizon=.8))

在相同 typed action contract 下，覆盖 interpolation、motion planner、learned primitive 与不同 queue mutation。

HANDLER CANDIDATES

事件与快速响应

@on_force(threshold=12.0)
def guard(obs, queue):
    queue.truncate(after="inflight")
    queue.push(stabilize(obs))
    emit_event("force-guard")

覆盖 on-frame、force、tactile、timeout 与 tracker-loss；同时监督触发条件、状态寿命、queue 权限和退出条件。

normal rollout→inject slip / occlusion / delay→handler fires→queue mutation→verifier grades recovery
完整能力不等于每次都应该写 handler。

训练中要同时加入负例：没有低延迟需求时不安装 handler；handler 已过期时主动移除；简单 CONTINUE 足够时不改 queue。否则模型会用越来越复杂的 runtime 掩盖不确定性。

06 · CAP-derived curriculum

在每个可监督端点注错，
让恢复成为常规训练数据。

CaP-X / CaP-RL 的直接启发是：具身 coding agent 可以在执行反馈与可验证奖励下改进[8]。Code as Runtime 进一步把显式 trace 中的目标、观察、约束、调用、状态和恢复作为可注错端点，而不是只增加成功示范数量。

01目标

替换相似物体、交换指令、加入 distractor。

02观察

遮挡、模糊、相机失配、深度缺失。

03约束

改变禁入区、负载、速度与力上限。

04调用

单位错误、不可达 pose、timeout、工具失败。

05状态

滑落、碰撞、抓空、延迟与部分执行。

06恢复

比较继续、重试、换视角、降级与停止。

SHARED API仿真与真机共享 typed action contract

参数、单位、坐标系、timeout、失败码和取消语义保持一致，只替换执行后端。

TRUST LEVEL记录 verifier 类型与置信度

规则、仿真物理、真机传感器、视觉模型和模型自评不能被当成同等可靠标签。

SHORTCUT AUDIT专门寻找 simulator exploitation

隔离场景、mesh、随机种子与任务模板；检查 trace 是否读取了部署时不存在的字段。

07 · RL acceptance conditions

RL 是否有效，
看策略能否在执行中改善。

主要验收对象是 observation-only policy 的 on-policy 表现，而不是 teacher rollout 或离线 token loss。除了最终成功率，还要检查 verifier 校准、恢复覆盖、策略熵、runtime 成本和真机迁移是否一起改善。

Privileged gaporacle success 与 observation-only success 的差距是否持续缩小？
Intervention rate需要 Magic Offset、GT recovery 或 teacher patch 的比例是否下降？
Verifier calibration过程奖励是否真的预测最终成功，而非只奖励漂亮 trace？
Recovery coverage未见扰动下，Agent 是否能定位失败端点并选择合适恢复？
Runtime economy成功率相同时，是否减少无效 token、helper 调用和 handler 数量？
Real-robot transfer共享接口之后仍剩多少 perception、contact、latency 和 calibration gap？

这条 RL 路线的核心，是让 Agent 在与部署一致的 runtime 中持续产生、执行和修正 trace，再用可校准的 verifier 把环境后果归因到具体决策。teacher、仿真真值和 Magic Offset 都只是进入这个闭环的脚手架；真正被优化并最终部署的，是只依靠可见 observation 与可调用 runtime API 的 on-policy Agent。
