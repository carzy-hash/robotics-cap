Code as Runtime · Real-data Trace SFT · 04

把真机采集，
变成可学习的 Runtime Trace。

这篇文章讨论 Code as Runtime 的真实数据 SFT：如何把一次连续示教留下的多相机视频、关节、力、触觉和动作，转成一组主 Agent 可以自回归学习的 context → decision 样本。关键不是把轨迹均匀切片，而是重建每个 turn 当时可见的信息、正在消费的 Action Queue，以及可以由代码或 Action Chunk 表达的下一次决策。

01 · SFT projection objective

先保存发生过什么，
再构造可以监督的下一 turn。

原始数据是不可变的事实层；SFT 样本是其中一种带假设的因果投影。投影器要显式选择 Agent 的唤醒时刻、推理延迟、可见 observation、queue 状态与 target decision，最终形成部署同构的 context → Agent output，而不能把未来信息悄悄写回输入。

投影不是改写事实。

同一个 raw episode 可以产生多种 decision timing 或 code 表达，但每个合成字段都要保留来源、方法和置信度；任何来自 future frame、人工修正或仿真真值的信息，都只能存在于 teacher sidecar，不能伪装成策略输入。

02 · Canonical event ledger

先把不同频率的流，
对齐到一条事件账本。

视频帧、控制命令、机器人实际运动与接触反馈并不同时到达。投影前先保留各自的 source timestamp，再建立经过校准的 episode clock；否则一个看似合理的 turn 可能已经使用了尚未到达的图像。

03 · Pseudo-time replay

推理期间，
机器人并没有停下来。

同步切片会产生一种不存在的世界：Agent 看完一段动作的最终结果，再决定这段动作该怎么写。伪时间回放则沿示教轨迹推进；每次 Agent 被唤醒后，Action Queue 在推理延迟 d 内继续消费，直到新的 decision 真正可用。

01选择唤醒点

固定 stride、阶段事件和异常事件混合；不要只在“正确答案显然”时唤醒。

02采样推理延迟

从目标模型延迟分布采样 d，而不是假设 decision 瞬时产生。

03冻结执行前缀

推理期间已发送或已执行的动作不可撤回，只有剩余 queue tail 可被替换。

04随机化截断节奏

在偏离前、偏离点与偏离后混合截断，避免策略只学会单一时间模式。

05 · Worked projection

一条拿香蕉示教，
怎样变成六个训练 turn？

原始 episode 只有连续运动与 sensor 流。投影器结合阶段边界、replay 拟合、queue 余量和因果可见性，构造部署时可能真实发生的 decision trace。下表同时标出哪些内容来自采集，哪些内容是可审计的合成。

“合理的解释”还不够，必须可以重放。

每个 code-built chunk 都重新展开并与实际轨迹比较；每次 queue mutation 都检查 executed prefix 是否保持不变；每个 CONTINUE 都验证 queue 没有在下一轮前耗尽。无法通过这些检查的 turn 不进入高置信 SFT 集。

06 · From canonical trace to SFT pairs

一条 episode trace，
展开成多轮 SFT 样本。

每个 turn 都以 available_until 之前的 context 为输入，以 Agent 的 decision、program 与 queue mutation 为主要监督目标；tool result 和 executor event 进入下一轮上下文。数据不保存不可见的“完整内心思维”，只监督可观察、可执行、可验证的结构化决策。

CAUSAL BOUNDARYavailable_until

所有策略输入必须能证明在该时刻之前已经到达。

QUEUE CONSISTENCYqueue_before / after

使 append、truncate 与 replace tail 可以离线重放。

OBSERVABLE REASONINGdecision / program / events

监督外显决策与执行结果，而不是假定存在唯一隐藏 CoT。

PROVENANCErecorded / derived / projected

每个字段标明来自原始记录、确定性计算还是 teacher 合成。

PRIMARY VIEWSFT next decision

输入 causal context；监督 decision、program、tool call 与 queue mutation。

DERIVED VIEWPreference

从被过滤的候选中比较错误 queue 修改与更经济的 CONTINUE。

DERIVED VIEWVerifier

复用进度、约束、失败端点、恢复与最终结果标签。

07 · Filtering and acceptance

数据是否能训练，
先看它能否通过因果审计。

成功 episode 不自动等于好数据。真正危险的是“成功但理由错误”的 trace：使用未来帧、泄漏真值、把人类接管归给 Agent，或用无法重放的 code 解释动作。过滤器必须在训练前找出这些捷径。

Temporal leakage是否有任何 context ref 晚于 available_until？
Replay equivalencecode 展开后与采集轨迹的误差是否低于分段阈值？
Queue validity是否修改了已经执行或正在飞行、不可撤回的前缀？
Privilege leakageGT、人工修正与 after-frame 是否进入了策略输入？
Intervention attribution人类接管前后是否切段，并标清控制权归属？
Failure value失败是否包含可定位端点与后续恢复，而非只有 terminal fail？
Media integrity所有外部图像、sensor window 与 chunk 引用是否存在且可解码？
Projection confidence低置信合成字段是否被降权、送审或排除出 SFT？

这条数据管线的直接产物，是与部署 runtime 同构的多轮 SFT 样本：输入只包含当时真正可见的 context，目标只包含下一次可执行的 Agent decision。随后 RL 可以复用同一 schema 继续 rollout，但 Real-data SFT 的首要成功标准仍是：训练样本里的世界，与采集当时真实存在的世界保持一致。
