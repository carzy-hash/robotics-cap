# Robotics Researcher Review

## Strengths

- 三版都正确覆盖了现代机器人操作的关键组成：真实遥操作与跨机构数据、仿真数据、行为克隆/VLA、离散动作 token 与连续 Action Chunk，以及仿真评测后仍需真机验证。
- 对仿真边界的处理较稳健：明确指出接触、噪声和硬件差异无法被完全复制，并提醒 CAP 不应读取真机不可得的 privileged state。
- CAP 被描述为高层受限策略程序，而不是用语言模型直接输出关节控制；底层仍交给动作专家、运动规划器或控制器，这一系统边界合理且重要。
- 三版都把 CAP 的优势写成待验证假设，并提出同数据、同动作底座、同硬件预算的公平对照，而非把可解释性直接等同于性能提升。
- Candidate C 对模型形态、监督端点和仿真闭环的定义最精确；Candidate A 对非专业读者最完整；Candidate B 的“工单”类比最容易理解。

## Risks

- “主流方法主要/只问动作像不像专家”“中间过程多数藏在模型内部”容易把研究现状说得过窄。实际系统可能已有分层规划、价值/奖励学习、语言或视觉辅助目标、安全过滤器、成功检测器和人工编写的恢复逻辑。CAP 的区别应表述为把这些过程统一为可训练、可审计的显式策略表示，而不是首次引入检查或恢复。
- Candidate B 的“看录像，还是先写工单”形成了易懂但略失真的二选一。真实遥操作数据不仅是录像，还包含本体状态和控制动作；主流系统也并非完全没有任务分解与外围控制逻辑。
- “失败定位可定位到目标、观察、约束、调用或验证步骤”只有在端点标签、检查器和日志本身可靠时成立。显式步骤改善可观测性，但不自动保证正确归因；错误也可能跨模块传播。
- “危险动作有机会在执行前被发现”不能暗示安全保证。学习式审计器可能漏检，确定性约束也受感知和模型误差影响，真机仍需要独立的低层安全机制与急停边界。
- “高层程序更容易跨机器人复用”仍是设计假设。不同 embodiment 的可达性、感知、末端执行器、动作语义和时序差异都可能迫使高层策略一起修改。
- Evidence 没有直接支持 Candidate C 所称“部署后的成功与失败回流”是主流数据混合的通常组成，也没有为 CAP 的六类端点、AR 模型、DSL/审计器架构提供外部实证；这些应清楚标作本文提出的 CAP 设计。

## Must address

- 将所有概括限定为“以行为克隆/VLA 动作策略为代表的主流端到端路线”，并补一句：不少实际系统已有分层规划、安全检查或恢复模块，但缺少 CAP 所主张的统一显式策略监督接口。
- 把“主流方法在仿真里批量训练和统计成功率”扩为训练、数据生成、域随机化、系统辨识/调参、鲁棒性与基准评测；同时坚持仿真结果不能替代真机闭环验证。
- 把 CAP 的错误定位、恢复、迁移和安全优势统一改为“可能/有机会”，并明确其成立条件：端点定义正确、监督来源可追溯、检查器经过校准、接口语义跨平台一致。
- 明确动作层时标：策略程序只以较低频率选择目标、工具和恢复分支；高频闭环控制由现有控制器或动作专家完成。否则读者可能误以为 AR 推理位于实时伺服环中。
- 公平实验除共享视觉骨干、动作专家和预算外，还应匹配总参数/计算、观测权限与安全控制；分别报告仿真和真机结果、sim-to-real gap、端点标注成本、推理延迟及检查器误报/漏报。
- 不把模型自评、仿真 privileged state 或规则检查结果直接当作世界真值；所有记录必须注明监督来源和置信度，并由独立传感器或真机 outcome 复核关键声明。

## Recommendation

以 Candidate C 作为技术事实和实验设计的底稿，采用 Candidate A 的杯子任务叙事与 Candidate B 的“工单”类比制作面向大众的最终稿。最终稿应先公平呈现强主流基线及其已有外围模块，再把 CAP 定位为“显式、受限、可逐端点监督的策略层”这一待证设计。保留当前证据文件中的事实/设计主张分界，并避免宣称 CAP 已经提升成功率、安全性、恢复或跨机器人迁移。
