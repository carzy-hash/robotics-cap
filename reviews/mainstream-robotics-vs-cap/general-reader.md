# General-reader review

## Strengths

- **Candidate A** 从“把红杯放进托盘”开始，完整覆盖数据、模型、监督、运行、仿真和评测；章节顺序自然，末尾对照表能帮助读者重新拼起全貌。
- **Candidate B** 的“看录像 / 写工单”类比最容易记住，也最清楚地表达了 CAP 并非替换动作控制，而是在理解任务和执行动作之间加入可检查的过程。篇幅、段落长度和叙事速度最适合普通读者。
- 三版都诚实写出了 CAP 的成本和待验证之处，没有把“更可读”直接说成“已经更强”；共享红杯案例也让两条流水线具备可比性。
- **Candidate C** 对模型形态、六类监督端点和仿真闭环定义最精确，适合作为技术附录或工程读者版本。

## Risks

- 三版都在首次出现时集中抛出 `VLA`、`token`、`Action Chunk`、`Flow Matching`、`AR`、`DSL` 等术语。A 虽解释了部分概念，但模型名与术语密度仍会让普通读者误以为理解这些名词是理解主线的前提。
- **Candidate A** 信息最全，但九节加表格容易产生“教科书目录感”；读者到第五节才进入 CAP，可能在完成主流流水线铺垫前失去注意力。
- **Candidate B** 最易读，但“工单”可能让人误解为 CAP 只生成文字清单，或误以为主流系统完全没有规划、约束检查和外围恢复逻辑。
- **Candidate C** 默认读者理解“输入输出契约、骨干、损失、隐状态、outcome、privileged state、embodiment”等概念，无法满足“任何人都能听懂”的目标；两条流水线虽结构对称，却需要较多背景知识才能比较。
- 当前对照多集中在组件和监督位置，缺少一段从同一条红杯任务出发、逐步并排展示两套系统在每一步分别做什么的直观比较。

## Must address

- 以 **Candidate B** 为正文基线，并在首次出现时用日常语言解释或删去缩写：VLA 是“看画面、听指令并输出动作的模型”；Action Chunk 是“一小段连续动作”；AR 模型必须给出全称或直接称“策略模型”。`token`、`Flow Matching` 等实现细节可移到注释或扩展阅读。
- 在“工单”第一次出现后立即说明：它是机器可执行、权限受限的策略步骤，不是给人看的普通文字，也不直接控制每个关节。
- 避免把主流路线绝对化。明确说明实际机器人系统也可能有规划器、安全规则和恢复模块；CAP 的主张是把这些关键决策统一暴露为可记录、可训练和可评测的监督端点。
- 增加一个紧凑的并排流程，用同一个红杯例子比较：`观察 → 直接预测短动作 → 执行 → 看最终结果` 与 `观察 → 生成受限策略 → 分步检查 → 调用短动作 → 验证/恢复`。随后再解释数据、监督和仿真的差异。
- 模型和数据集名称不应承担论证。正文只需各举一两个例子，其余放到“扩展阅读”；同时为“监督端点”补一句白话定义：系统在过程中可以被检查、纠正并留下记录的位置。
- 保留公平实验与局限段落，但把指标翻译成读者可感知的问题：能否完成、出错后能否救回来、是否更安全、排查需要多久、换一台机器人要改多少。

## Recommendation

选择 **Candidate B** 作为 canonical 版本，吸收 Candidate A 中“数据从哪里来、五类评测指标、仿真可能作弊”的简明解释，并把 Candidate C 保留为技术附录。正文应坚持一个红杯案例贯穿到底，先用两条并排流程建立地图，再展开数据、模型、监督、仿真与评测；这样普通读者既能记住核心差异，也不会把 CAP 误解成用大语言模型直接控制机器人。
