# ML Systems Engineer Review

## Strengths

- 三稿都正确区分了高层策略与低层控制：CAP 并非让语言模型直接输出关节命令，而是让受限策略调用运动规划器、控制器或动作专家。这个边界对于工程可行性和安全论证都很重要。
- Candidate A 覆盖最完整且仍易读，清楚解释了数据来源、VLA/Action Chunk、监督位置、仿真作用及公平对比原则，适合作为面向普通读者的主干。
- Candidate B 的“工单”类比最有效，能让非专业读者理解显式目标、检查和恢复为何比只看最终动作更容易诊断。
- Candidate C 最接近可实现的系统规格：明确提出感知、AR 策略、审计器、动作执行四种职责，并用受类型与权限限制的 DSL/工具调用序列表达策略，而非任意代码。
- Candidate C 给出的六类监督端点、监督者来源、仿真端点注错、禁止 privileged state，以及共享视觉骨干和动作专家的公平实验约束，构成了可证伪的工程假设。

## Risks

- 三稿都还没有定义最小数据 schema。若不固定 episode/step 标识、时间戳、坐标系、观测版本、策略版本、工具调用及参数、返回值、监督者类型与置信度、恢复分支和最终 outcome，所谓“过程监督”很容易退化成不可复现的自然语言日志。
- “监督端点”混合了三类性质不同的信号：可由规则确定的约束、带噪声的感知判断，以及依赖任务语义的策略选择。若将它们作为同等可靠的标签训练，模型自评或视觉误判可能被当成物理事实。
- 策略程序与动作接口的契约仍偏概念化。缺少参数类型、单位、坐标系、前置/后置条件、超时、幂等性、可取消性、失败码和安全仲裁优先级，难以保证仿真与真机共享同一执行语义。
- 训练闭环只说“日志回流训练”，尚未说明分别训练策略生成器、审计器和动作专家时，标签如何产生、哪些样本进入哪个模型、如何避免同一模型生成并认可自己的标签，以及离线数据与在线失败数据怎样采样。
- CAP 增加审计和多次观察后，会改变控制频率、任务耗时与可用计算量。若基线只按成功率配平而不配平动作调用次数、观察预算、墙钟时间和算力，比较会偏向 CAP；反之只严格配平延迟也可能掩盖安全检查的价值。
- Candidate A/B 对“跨机器人复用”的表述虽有保留，但仍可能被读成已证实优势。高层 DSL 只有在对象语义、技能集合、坐标契约和失败码保持兼容时才可迁移，底层技能覆盖差异仍可能要求重写策略。

## Must address

- 在正文中加入一个紧凑但明确的最小轨迹记录示例，例如 `observation → policy_step → audit_result → action_call → action_result → verification → recovery → outcome`，并指出每项都带时间、版本、来源和置信度；不必展示完整 JSON，但必须让读者知道数据不是自由文本。
- 明确监督可信度层级：碰撞/速度/权限等由确定性检查器裁决，接触与成功状态优先使用传感器或仿真物理结果，视觉判别与模型自评只能作为带置信度的估计；安全控制器可以覆盖策略程序。
- 明确仿真接口与真机接口共用同一 typed action API，仿真只替换后端；策略可见字段必须取两者交集，privileged state 仅可供评测器生成标签，不得进入策略输入。
- 补全训练闭环：原始示范初始化动作专家和策略；扰动运行产生端点标签与恢复样本；规则/物理标签可自动回流，感知或语义标签需人工抽检；训练、审计与评测数据按场景和对象隔离，避免泄漏。
- 公平评测至少报告两组结果：等原始示范/真机预算下的端到端效果，以及等墙钟延迟/动作与观察预算下的系统效果。除成功率外，预先定义恢复率、约束违规率、诊断时间、端点标注成本、p50/p95 推理延迟、迁移工时和 sim-to-real drop。
- 将“CAP 的优势”统一写成待验证假设，尤其是安全、诊断和跨 embodiment 迁移；避免把可读性直接等同于正确性或可控性。

## Recommendation

以 **Candidate A** 作为公开文章主体，因为它在“任何人都能听懂”和流程完整性之间最好；吸收 **Candidate B** 的“工单”类比作为开场与贯穿隐喻；将 **Candidate C** 中四模块架构、typed DSL、六类端点、privileged-state 边界和公平实验设计压缩成一个“工程上如何落地”的专栏或附录。

发布前必须补齐最小数据记录、动作接口契约、监督可信度和训练回流四处定义。否则文章能解释 CAP 的动机，却还不足以让读者判断它是否可以真实实现与公平验证。
