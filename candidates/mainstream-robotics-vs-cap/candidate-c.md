# 从 VLA 到可监督策略程序：主流机器人操作栈与 CAP

本文用同一个输入输出契约比较两类系统。任务是“把红杯放进托盘”；输入是多视角图像、语言指令和机器人状态；最终输出都必须落到受约束的机器人动作。差异在于策略如何表示，以及训练信号能进入哪些位置。

## 主流操作栈

### 数据层

主流数据混合通常包含四类：人类遥操作的真实轨迹、跨实验室标准化轨迹、仿真示范或自动生成轨迹、部署后的成功与失败回流。DROID 展示了分布式真实遥操作；Open X-Embodiment 展示了跨 22 种机器人的统一数据；RoboCasa 和 Isaac Lab 展示了大规模仿真训练能力。

一个训练样本通常包含观察、语言、机器人本体状态和动作序列。语言标注质量、相机标定、动作坐标系与机器人差异，会直接影响数据能否混合。

### 模型层

VLA 用视觉语言骨干提取语义，再输出机器人动作。RT-2 和 OpenVLA 代表动作 token 路线；π0 使用 VLM 骨干加 Flow Matching 动作专家，生成连续 Action Chunk。实际系统还可能拆分高层规划与低层控制，但动作预测仍是训练闭环的核心出口。

### 监督层

行为克隆的核心损失作用在专家动作上。系统也可加入成功标签、奖励、价值函数或辅助视觉语言目标。困难在于目标选择、观察充分性、约束满足和恢复意图通常没有统一的显式标签，它们被隐式压入模型状态或由外围逻辑处理。

### 执行与评测层

部署循环是 `observe → infer action chunk → execute → observe`。仿真承担大规模回放、训练、域随机化和基准评测；真机承担最终验证。典型指标包括任务成功率、分布外泛化、动作效率、碰撞/超力、安全停止和失败恢复。

## CAP 操作栈

### 模型形态

CAP 把模型分成四个职责：

1. 感知模块把视觉与状态整理为可查询观察；
2. AR 策略模型生成受限策略程序；
3. 审计器检查目标、信息、约束和调用权限；
4. 动作专家或传统控制栈执行短 Action Chunk。

这里的策略程序不是任意 Python，也不是关节控制器。它更接近受类型和权限限制的 DSL 或工具调用序列，例如 `locate(red_cup)`、`request_view(left)`、`grasp(target, force_limit)`、`verify(in_gripper)`。

### 数据形态

CAP 可以直接复用现有图像—语言—动作数据，但希望把执行轨迹再结构化为：目标对象、当前假设、观察请求、约束、工具调用、接口返回、验证结果和恢复分支。人工示范提供“怎样做”，仿真扰动和部署失败提供“什么时候不该继续”和“如何恢复”。

### 监督端点

一条策略执行至少暴露六类端点：

- **目标端点**：对象和成功条件是否正确；
- **观察端点**：证据是否足够，是否需要换视角；
- **约束端点**：碰撞、速度、力、区域和权限是否满足；
- **调用端点**：动作接口的参数和返回值是否合法；
- **状态端点**：执行后的世界是否符合预期；
- **恢复端点**：继续、重试、降级和停止是否合理。

监督者可以是确定性规则、几何/物理检查器、视觉判别器、人类标签或真机传感器。关键是记录监督者类型与置信度，不能把模型自评当作物理事实。

### 仿真闭环

CAP 在仿真中运行完整策略解释器，而不是只运行最终动作网络。仿真器向策略暴露与真机对齐的观察和动作接口；测试框架在每个端点注入对象位姿、遮挡、相机噪声、摩擦、执行失败和延迟扰动；轨迹记录器保存策略版本、调用、世界状态、检查结果和最终 outcome。

这些记录可以训练策略生成器、审计器和恢复选择器。为防止 simulator exploitation，策略不得读取真机不存在的 privileged state；评测还需要未见场景和真机闭环。

## 同约束对比

| 维度 | 主流 VLA / 动作策略 | CAP |
| --- | --- | --- |
| 策略表示 | 隐状态 + 动作输出 | 显式受限策略程序 |
| 动作出口 | token 或连续 chunk | 相同动作底座，由程序调用 |
| 监督密度 | 以动作和 outcome 为主 | 动作 + 六类过程端点 |
| 错误归因 | 轨迹级或事后分析 | 可定位到程序步骤与接口 |
| 仿真用途 | 数据、训练、成功率、鲁棒性 | 再加入端点注错和恢复训练 |
| 跨 embodiment | 依赖动作归一化与微调 | 高层程序潜在复用，接口仍需适配 |
| 新增成本 | 大规模轨迹数据 | 程序 schema、审计器、过程标签、推理延迟 |

## 公平实验

CAP 必须与强 VLA 基线共享视觉骨干、动作专家、示范数据、任务集合、仿真随机种子和真机预算。除了成功率，还应报告：扰动恢复率、约束违规率、错误定位时间、每任务新增标签成本、策略推理延迟、动作接口迁移工作量，以及仿真到真机的性能下降。

CAP 的可证伪假设是：在不显著牺牲基础成功率的前提下，显式策略与端点监督能提高恢复率、降低安全违规与诊断成本，并让高层任务逻辑更容易跨硬件复用。若这些指标没有改善，增加的系统复杂度就不值得。
