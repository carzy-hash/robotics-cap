机器人是怎样从示范里学会动作的

在比较不同机器人路线之前，先把今天最常见的训练方法看清楚：人遥操作机器人完成任务， 系统记录机器人当时看到了什么、听到了什么、身体处在什么状态，以及人让它做了什么动作。 模型学习的核心，就是从前三项预测第四项。

先建立一个最重要的直觉：输入是什么，输出是什么

任务是“把薯片袋放到白色盘子里”。机器人并不是直接理解一个完整世界。 它在当前时刻拿到几组信息，再由策略模型产生接下来一小段动作。

数据怎样采集：人先操作给机器人看

最容易理解、也最接近真实机器人的数据采集方式，是让人遥操作一台真实机器人。 人看着桌面，用控制器带机械臂靠近目标、抓住物体并移动到指定位置。

系统不是只录一段供人观看的视频。它会沿着同一条时间轴保存相机画面、任务指令、机器人身体状态和人的控制动作。 因此每一个时刻都像一道有标准答案的题：模型拥有当时的输入，而人类动作就是答案。

BC 在训练什么：反复做“看状态，猜动作”

BC 是 Behavior Cloning 的缩写，中文通常叫行为克隆。训练时，系统先把人类动作遮住， 只把图像、Instruction 和 Robot State 交给模型。模型预测 Action Chunk 后，训练器再揭晓人类示范，比较两者相差多少。

如果模型预测机械臂应该向左移动，而操作者实际向右移动，训练算法就调整模型参数。 换一条记录继续重复后，模型逐渐学会在相似画面和任务下给出接近示范的动作。

BC 的主要监督对象是动作。老师最直接检查的是：模型输出的 Action Chunk，和人类示范动作有多大差异。

训练完成后，机器人怎样真正运行

部署时已经没有人类动作作为标准答案。机器人读取当前图像、Instruction 和 Robot State， 由 VLA 策略模型生成一个 Action Chunk，执行后重新观察，再生成下一段动作。 

看当前状态 → 输出短 Action Chunk → 执行 → 重新观察 → 输出下一段动作

Action Chunk 不是所有电机细节。它表达接下来一小段时间的局部动作目标； 更高频、更平滑的电机控制仍由底层控制器完成。

先理解它的价值，再理解它的边界

BC 的价值是直接：人类操作天然提供清晰的动作答案，不需要为每条示范额外写一篇详细计划。 只要采集足够丰富、质量足够好的操作记录，模型就可以不断学习新的动作模式。

但动作答案没有完整写出“为什么这样动”。机器人伸错方向时，原因可能是目标理解错、图像没看清、路径不合适， 也可能只是最后控制没抓稳。实际系统可以在外围加入规划、安全规则和恢复模块， 但这些中间判断不一定统一进入与动作同样清晰的训练记录。

到这里，才开始看 CAP 增加了什么

CAP 不替换 BC 已经学到的动作能力。相机继续提供观察，VLA 或其他动作模型继续产生 Action Chunk， 底层控制器继续负责高频执行。

CAP 想增加的是动作之前和之后的策略层：目标是谁、当前信息够不够、要满足什么约束、 动作后应该检查什么、失败后应该重试还是停止。

这里的 CAP 是本项目对“用可执行策略程序连接任务理解与机器人动作”这条路线的名称。 策略通常由 AR（Autoregressive，自回归）模型逐步生成；简单理解就是像写代码一样，一步接一步写出目标、调用、检查和恢复逻辑。

用同一个薯片任务走一遍 CAP Loop

第一轮观察发现薯片袋和盘子，但夹爪离目标还远。策略模型生成“靠近薯片”的局部调用； 检查器确认目标可见、路径没有明显冲突后，底层动作模型执行一个短 Action Chunk。

第二轮重新观察，机器人发现夹爪已经来到薯片旁。策略变成“抓取并验证”；动作执行后， 系统检查薯片是否真的随夹爪抬起。如果没有，就留下“抓取验证失败”的记录，选择重新定位、重试或停止， 而不是假设上一段动作一定成功。

CAP 的训练记录比动作多了什么

原来的图像、Instruction、Robot State 和 Action Chunk 仍然保留。CAP 额外记录当前目标、策略步骤、 检查结果、动作接口返回值、验证结果和恢复选择。这样一次失败不只剩“最后没完成”， 还可能明确停在“目标看错”“观察不足”“动作调用失败”或“执行后验证失败”。

CAP 并不自动保证策略正确或机器人安全。它提供的是更多可以检查和训练的位置； 是否真的提升成功率、恢复能力和诊断效率，仍需要与相同动作底座做公平实验。
