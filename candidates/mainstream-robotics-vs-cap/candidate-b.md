# 两种教机器人干活的方法：看录像，还是先写工单

想象一个新员工要学会“把红杯放进托盘”。第一种培训方式，是让他看几万段熟练工人的操作录像，然后在看到相似桌面时模仿相似动作。第二种方式，是除了看录像，还要求他先写一张简短工单：目标是谁、先检查什么、哪些地方不能碰、失败后怎么办。主流机器人学习更接近第一种；CAP 想把第二种也加入流水线。

## 主流流水线：从示范直接学动作

真实机器人数据通常由人遥操作产生。操作者拿着控制器完成抓取、开抽屉或叠衣服，系统同步保存图像、指令、机器人状态和动作。DROID 是这种真实数据的代表。Open X-Embodiment 则把多个机构和多种机器人的数据整理在一起。数据不够时，团队还会在 LIBERO、RoboCasa 或 Isaac Lab 这样的仿真环境里收集示范、批量试错和测试扰动。

模型的工作，是从“现在看到什么、用户要什么”预测“接下来怎样动”。RT-2、OpenVLA 一类 VLA 可以把动作变成 token；π0 一类模型可通过连续动作专家生成平滑 Action Chunk。它们像不同类型的熟练工：表达动作的方法不同，但都尽快把观察变成可执行动作。

训练时最常见的老师是专家动作。模型伸手方向越接近示范，损失越小。评测时则让机器人真正做任务，看成功多少次，换物体、光线、场景后是否还能做，以及会不会碰撞、卡住或无法恢复。

这条路线直接、有效，也已经形成了大规模数据和模型生态。它的困难是：录像告诉了机器人“手怎样动”，不一定明确告诉它“为什么要这样动”。红杯拿错时，错误可能来自识别错目标、观察不够、路径不安全，也可能只是最后控制没抓稳。所有原因最后都表现为“任务失败”。

## CAP 流水线：先形成可检查的策略工单

CAP 保留同样的摄像头、机器人数据、动作模型和仿真器，但在“理解任务”和“执行动作”之间加入受限策略程序。红杯任务不再只剩一串动作，而会留下这样的执行记录：

> 找红杯 → 检查是否看清 → 选择抓取点 → 检查禁入区 → 执行短动作 → 确认抓稳 → 移到托盘 → 确认完成。

高层 AR 模型负责产生这张工单；视觉模块提供观察；运动规划器、控制器或连续动作专家仍负责把局部目标变成平滑动作。CAP 不是用文字代替控制，而是让不同模块的职责更明确。

工单上的每一栏都可以成为监督端点。目标错了，在执行前就能拦下；画面看不清，可以要求换视角；抓取失败，动作接口会返回失败；最终状态不对，程序进入重试或停止。于是训练数据不只记录“正确动作”，也能记录“错误出现在哪里”和“系统如何恢复”。

## 仿真不只是考场，也成为质检线

主流方法在仿真里批量训练和统计成功率。CAP 还可以对工单逐项质检：遮住杯子，看程序是否先补观察；改变摩擦力，看抓取失败后是否重试；移动托盘，看程序是否重新定位；给出不安全路径，看约束检查是否拒绝。

一次运行会留下观察、策略、检查、动作和结果。失败样本可以直接回到训练流程。但仿真不能成为答案生成器：策略只能读取真机也能获得的信息，最后仍要在真实硬件上验证。

## 差别不在“有没有大模型”，而在监督颗粒度

| 流水线 | 主流方法 | CAP |
| --- | --- | --- |
| 输入 | 图像、指令、机器人状态 | 相同 |
| 中间过程 | 多数藏在模型内部 | 显式策略程序 |
| 输出 | 动作 token / Action Chunk | 策略调用短动作接口 |
| 老师检查什么 | 动作像不像示范、任务是否成功 | 还检查目标、约束、观察和恢复 |
| 失败后 | 从最终轨迹倒查 | 从具体监督端点定位 |
| 最大代价 | 数据贵、黑箱失败 | 系统更复杂、推理更慢、端点需要设计 |

CAP 的潜在优势，是让一次失败变得更有信息，让策略层更容易跨机器人复用，也让危险动作有机会在执行前被发现。它的风险同样明确：程序可能写错，检查器可能误判，接口工程会增加成本，长思考也不适合每一种高速任务。

因此验证不能只展示一段成功视频。两条流水线必须使用相同任务、数据、动作底座和硬件，比较成功率、扰动恢复、安全违规、诊断时间和迁移成本。只有当这组指标整体改善，“先写工单”的优势才成立。
